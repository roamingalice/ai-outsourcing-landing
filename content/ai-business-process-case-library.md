# AI 商業流程架構案例庫

更新日期：2026-05-07

## 使用定位

這份文件是 Alice / ASify / Frank 共同累積 AI 商業流程案例的母資料源。

用途：

- 對外服務頁精選案例
- 講座與簡報素材
- 接案前診斷與提案
- 社群內容與文章
- 內部複盤與能力地圖

核心主軸：

AI 不是單點工具，而是把品牌的知識、流程、客戶訊號、權限風險與營運回饋，設計成可以持續運作、測試、糾錯、成長的商業系統。

## 維護原則

每個案例都要同時保留「內部版」與「可公開版」。

- 內部版：可以記錄真實技術細節、踩雷、修正、限制、敏感背景。
- 可公開版：保留商業價值與方法論，不揭露 token、API key、客戶資料、真實營收、供應鏈細節、合作名單。

每次新增案例，至少回答：

1. 這解決誰的什麼痛？
2. 原本流程卡在哪裡？
3. AI 做哪一段？人在哪裡把關？
4. 怎麼測試成功？怎麼發現錯誤？怎麼回報與修正？
5. 這個案例能轉譯給哪些受眾？
6. 是否適合放到服務頁或講座裡？

## 報告參考：2026 台灣產業 AI 化大調查

這份報告只作為市場語言參考，不照單全收。

可借用的洞察：

- AI 已從選配變標配，企業關心的問題已從「要不要用」轉向「怎麼落地」。
- 多數企業仍停在個人工具使用，真正完全整合進公司常規流程的比例很低。
- Shadow AI 與治理落後是主要風險，效率不能用資安與失控換來。
- 零售服務業成長快，但常停在內容創作、社群行銷等單點應用，尚未進入組織流程。
- 企業缺的不是單一工具，而是能拆解問題、整合流程、資料、權限、AI 與驗證的 Solution Architecture 能力。

對 Alice × Frank 的定位幫助：

Alice 負責把商業問題翻譯成流程與規格，Frank 負責把規格做成可運作、可部署、可維護的系統。這正是報告中提到企業缺乏的中間層。

## 初始案例

### 1. ASify 客服知識庫 / 導購 AI

**狀態**
ASify production / 可做講座案例 / 可衍生服務

**來源**
ASify

**能力分類**
- 客服 / 售後 / 服務流程
- 品牌知識庫 / AI 分身
- 權限 / 資安 / 風險控管
- 測試驗證 / Closed-loop / 使用者端驗證
- 教育訓練 / SOP 複製 / 組織落地

**真實場景**
ASify 客人常詢問尺寸、布料、褲長、身形推薦、退換貨、門市、物流、VIP 等問題。這些問題過去仰賴真人客服與 Alice 的品牌知識。

**原本流程**
客服每次依記憶或資料頁回答，常見問題重複出現，新人訓練成本高，品牌語氣與禁忌用語容易不一致。

**AI 解法**
把品牌資料、FAQ、尺寸表、布料說明、退換貨政策、語氣規則與禁忌用語整理成 AI 可讀的知識庫。AI 回答時必須依據資料庫，不確定時轉真人客服。

**技術組成**
Markdown 品牌資料庫、客服規則 prompt、尺寸 / 布料 / 產品導購資料、真人轉接 gate。

**人工 gate**
涉及訂單、庫存、客訴、特殊退換貨、折扣承諾時轉真人。

**Closed-loop 設計**
- 測試：用常見問題與刁鑽問題測回答是否符合品牌語氣與政策。
- 驗證：檢查是否附正確連結、是否主動追問身高體重、是否避免禁忌詞。
- 回報：把答不出來或高風險問題回收成知識庫補洞。
- 糾錯：更新 FAQ、禁忌語、產品資料與轉真人條件。
- 持續成長：客服問題累積後可轉成內容、FAQ、訓練素材。

**資料與權限風險**
不應讓 AI 自行查詢完整訂單、金流或私人資料；對客承諾必須保守。

**資安 / 風險設計**
資料最小化、品牌禁忌、真人 gate、不得編造不存在產品與優惠。

**商業效果**
降低客服重複工作、提升新人一致性、把客戶問題轉成內容與銷售素材。

**適合受眾**
美業、零售、電商、連鎖店、教育訓練、客服團隊。

**可轉譯說法**
美業：把店長腦袋裡的服務經驗變成 AI 可查、員工可學、客人可感受到一致溫度的知識庫。

**可公開程度**
可公開，但不公開完整內部 prompt 與敏感政策。

**是否已上服務頁**
部分已用 ASify 9 agents 方式提及。

**下一步可補強**
加入真實對話前後對比、客服問題分類量化、常見問題轉內容的範例。

### 2. ASify Shopline 退換貨自動化 Bot

**狀態**
ASify production / 可做講座案例 / 可放服務頁候選

**來源**
ASify

**能力分類**
- 營運自動化
- 客服 / 售後 / 服務流程
- 跨系統整合 / Solution Architecture
- 權限 / 資安 / 風險控管
- 測試驗證 / Closed-loop / 使用者端驗證

**真實場景**
客服同事需要查 Shopline 訂單並建立退換貨單。這類工作重複、細節多、容易填錯。

**原本流程**
人工登入後台、查訂單、複製資料、填退換貨、處理折扣與出貨資訊。

**AI 解法**
Telegram Bot 收訂單編號或截圖，查詢 Shopline 後台，回傳客人資訊與訂單連結，再用對話狀態機引導輸入退換貨資料，自動建單、折扣歸零、出貨、截圖 QR code。

**技術組成**
Telegram Bot、Playwright 操作 Shopline 後台、共享 browser session、Claude OCR、對話狀態機、任務佇列。

**人工 gate**
客服仍需提供/確認退換貨資料；特殊客訴或異常狀態不應全自動放行。

**Closed-loop 設計**
- 測試：用測試訂單、截圖、不同物流與退換貨情境跑流程。
- 驗證：檢查 Shopline 後台是否真的建單成功、折扣是否歸零、QR code 是否可用。
- 回報：Telegram 回傳結果與截圖。
- 糾錯：session 過期、後台 UI 變動、物流欄位變動時需通知並修正。
- 持續成長：把常見退換貨情境變成更清楚的 SOP 與例外處理。

**資料與權限風險**
涉及訂單、客資、物流資訊與後台 session。不能公開 token、session、訂單截圖。

**資安 / 風險設計**
限制入口在內部 Telegram；高風險異常人工處理；session 過期通知；不對外暴露後台。

**商業效果**
降低填錯率、節省客服時間、讓售後流程可複製。

**適合受眾**
電商、零售、美業預約/改期/售後、連鎖服務業。

**可轉譯說法**
美業：預約改期、售後照顧、回訪提醒、客訴資料都可以做成半自動流程，不靠員工每次從頭想。

**可公開程度**
可公開概念與流程；技術細節可講到 Playwright / Telegram / OCR，但不公開客資與後台畫面。

**是否已上服務頁**
服務頁提到退換貨 bot。

**下一步可補強**
補一張「人工流程 vs Bot 流程」圖。

### 3. ASify KOL / 老師合作管理系統

**狀態**
ASify production / 可做講座案例 / 可衍生服務

**來源**
ASify

**能力分類**
- KOL / KOC / 推薦合作
- 營運自動化
- 內容生產 / 社群發布
- 內部工具 / 人脈與組織管理
- Dashboard / 營運總覽 / 決策回饋

**真實場景**
ASify 有瑜伽老師、KOL、KOC 合作，需要管理名單、聯繫、合作條件、開團日期、推薦碼與分潤。

**原本流程**
靠人工記憶、零散 Sheet、信件與訊息往返，合作狀態容易漏追。

**AI 解法**
以 Telegram Bot 為指令入口，一張 Master Sheet 管理合作生命週期；Apify 爬 IG 資料，Gmail 產邀請草稿與偵測回信，Google Calendar 同步開團日期，Claude 做語意分析，sheet monitor 定期偵測狀態變化。

**技術組成**
Telegram Bot、Google Sheet、Gmail、Google Calendar、Apify、Claude CLI、排程監控。

**人工 gate**
不直接發 IG DM；由 Alice 手動發送或確認重要訊息。合作條件與分潤需人確認。

**Closed-loop 設計**
- 測試：檢查新增 KOL、狀態變更、邀請草稿、回信偵測、日期同步。
- 驗證：Sheet 狀態是否正確、Calendar 是否有事件、合作條件是否一致。
- 回報：Telegram 推播待處理事項與異常。
- 糾錯：回信分類錯誤、狀態漏改、開團日期錯誤時回寫修正。
- 持續成長：累積合作轉換率、回信話術、KOL 類型與成效。

**資料與權限風險**
涉及合作名單、分潤、信件內容、推薦碼。對外需匿名。

**資安 / 風險設計**
OAuth 權限集中管理；重要對外訊息人工確認；不自動發 DM。

**商業效果**
讓合作漏斗可視化、可追蹤、可複製；合作從「有空再追」變成系統。

**適合受眾**
美業、零售、品牌、電商、KOL 經營、連鎖店異業合作。

**可轉譯說法**
美業：熟客、設計師、老師、KOC、異業合作都可以變成推薦節點，不能只靠店長記憶。

**可公開程度**
可公開架構與流程；不公開名單與分潤明細。

**是否已上服務頁**
主服務頁有 KOC 推薦與 KOL 自動開發。

**下一步可補強**
補合作漏斗指標：爬取、聯繫、回覆、開團、成交。

**原本流程**
好評躺在 Google、商品頁、社群或客服訊息中，很難被系統性使用。

**AI 解法**
用 scraper / 匯入方式收集評論，寫入資料庫，再按主題、產品、身形、布料、服務體驗整理成內容素材、FAQ、成交話術與策略訊號。

**技術組成**

**人工 gate**

**Closed-loop 設計**
- 測試：抓取是否完整、去重是否正確、分類是否合理。
- 驗證：抽樣比對原始評論與整理結果。
- 回報：產出主題摘要與可用素材。
- 糾錯：錯分、重複、敏感內容移除。
- 持續成長：好評變成商品頁、社群、客服知識庫與訓練素材。

**資料與權限風險**
涉及顧客評論與可能可識別資訊；公開需匿名或遵守平台規範。

**資安 / 風險設計**
保留原始來源；公開版去識別；敏感評論不直接外用。

**商業效果**
把客戶信任轉成可重複使用的成交素材與服務改進訊號。

**適合受眾**
美業、零售、電商、餐飲、連鎖服務業。

**可轉譯說法**
美業：客人稱讚你的話，是最有價值的成交素材，不應只躺在 LINE 或 IG 私訊裡。

**可公開程度**
可公開方法，引用需挑選與匿名。

**是否已上服務頁**
尚未明確上服務頁。

### 6. ASify 工廠群組監聽 Bot

**狀態**
ASify production / 可做講座案例

**來源**
ASify

**能力分類**
- 營運自動化
- 資料治理 / 資料整合
- Dashboard / 營運總覽 / 決策回饋
- LINE Bot / LIFF / 互動工具
- 權限 / 資安 / 風險控管

**真實場景**
工廠 LINE 群組有大量文字與圖片訊息，包含修改、進度、照片、確認事項。重要資訊容易被洗掉。

**原本流程**
人工翻群組、截圖、記錄，容易漏掉細節。

**AI 解法**
LINE webhook 監聽群組文字與圖片，圖片上傳取得 URL，訊息寫入 Notion。MiniMe / AI 後續讀 Notion 彙整重點。

**技術組成**
LINE webhook、Vercel Serverless Function、圖片上傳、Notion DB、定期 AI 彙整。

**人工 gate**
Bot 純監聽不回話，不操作其他系統。

**Closed-loop 設計**
- 測試：群組文字與圖片是否完整寫入。
- 驗證：Notion 欄位、圖片 URL、時間與來源是否正確。
- 回報：週期性彙整重點給 Alice。
- 糾錯：漏訊息、圖片失效、Webhook 異常。
- 持續成長：累積成供應鏈進度與產品修改知識庫。
